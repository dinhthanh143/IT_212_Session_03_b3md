@Service
public class CartService {
    @Autowired
    private CartRepository cartRepository;
    @Autowired
    private ProductRepository productRepository;
 
    public void addToCart(Long userId, Long productId, int quantity) {
        Product product = productRepository.findById(productId)
            .orElseThrow(() -> new ResourceNotFoundException("Product not found"));
        
        CartItem item = cartRepository.findByUserIdAndProductId(userId, productId);
        if (item != null) {
            item.setQuantity(item.getQuantity() + quantity);
        } else {
            item = new CartItem(userId, productId, quantity);
        }
        cartRepository.save(item);
    }
}

## Nhận xét thẳng thắn về đoạn code `CartService.java` hiện tại

Để bạn có thêm góc nhìn phản biện (đúng tinh thần thẳng thắn), đoạn code hiện trường của nhân sự bên bạn đang dính **3 lỗi chí mạng** về mặt hệ thống mà một System Analyst phải nhìn ra ngay lập tức:

1. **Thiếu kiểm tra tính hợp lệ của `quantity`:** Khách hàng hoàn toàn có thể truyền `quantity = -5` thông qua API. Hệ thống sẽ lấy số lượng hiện tại trừ đi 5. Đây là lỗi logic kinh điển tạo ra lỗ hổng "Hack số lượng/Giảm giá tiền" trong giỏ hàng.
2. **Bỏ quên Inventory hoàn toàn:** Code chỉ kiểm tra xem sản phẩm có tồn tại hay không (`productRepository.findById`), chứ không hề lấy thông tin `product.getStock()` hay gọi qua `InventoryService` để check xem kho còn đủ hàng hay không. Khách có thể mua 1000 món trong khi kho chỉ còn 2 món.
3. **Race Condition (Xung đột đồng thời):** Ngay cả khi có check kho, đoạn code này cũng đang lấy dữ liệu lên, tính toán trên bộ nhớ (`setQuantity`) rồi lưu xuống (`save`). Nếu người dùng bấm click liên tục (hoặc dùng script gọi API đồng thời), hệ thống sẽ tính toán sai số lượng do không có cơ chế Locking (Pessimistic/Optimistic Locking).
